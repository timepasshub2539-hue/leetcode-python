MAIN ARTICLE

Introduction

Binary string problems show up constantly in coding interviews because they look deceptively simple — a string of 0s and 1s — while quietly testing whether you can spot structure instead of reaching for brute force. "Maximize Active Section with Trade I" is a great example. On the surface it asks you to simulate a single swap operation. Underneath, it's really testing whether you understand run-length thinking: the idea that consecutive repeated characters can be collapsed into a handful of numbers, and that most "simulate every possibility" problems secretly reduce to a single arithmetic pass.

This pattern isn't just an interview trick. The same idea — grouping runs of identical values and reasoning about their boundaries — powers things like free-space coalescing in memory allocators, run-length encoding in image compression, and disk block management. Learning to recognize it here will help you recognize it again in a dozen other problems.

By the end of this article, you'll understand why the brute-force approach breaks down at scale, how to reframe the problem in terms of "runs" instead of individual characters, and how to write a clean O(n) Python solution that never even needs to touch the actual string.

Problem Overview

You're given a binary string s made up only of '0' and '1' characters. Think of '1' as "active" and '0' as "inactive."

You may perform exactly one operation: choose a maximal group of consecutive 1s that is directly bordered by a 0 on both sides (a group that is "boxed in"), erase it, and then fill the entire span — the erased 1s and both bordering 0s — with 1s.

Your task is to return the maximum number of active (1) positions achievable after performing this operation at most once. If no valid group exists, the answer is simply the count of 1s already in the string.

Example

Take s = "1000100".

  • Original active count: 2 (the two 1s).
  • There's a group of 1s in the middle ("1000100" → index 4), boxed in by three 0s on the left and two 0s on the right.
  • Trading it: erase that group, fill the whole span (both zero-runs plus the middle 1) with 1s.
  • Result: "1111111" → 7 active positions.

Second example: s = "0100".

  • Original active count: 1.
  • The single 1 at index 1 is boxed in by one 0 on the left and two 0s on the right.
  • Trade it: the whole span becomes 1s → "1111" → 4 active positions.

Every output makes sense once you realize the operation doesn't really "erase and refill" — it just merges a 1-block with its two neighboring 0-blocks into one long run of 1s.

Intuition

Here's the mental shortcut that unlocks this problem: the block of 1s you erase gets turned right back into 1s by the same operation. It cancels itself out completely. Nothing about that middle block changes the final count.

So what actually changes? Only the 0s immediately to the left and right of that block — those flip from inactive to active. That's the entire gain.

Picture it as a sandwich: the 1-block is the filling, and the 0-runs on either side are the bread. Trading turns the bread into filling too. Your gain from any single trade is just (size of left zero-run) + (size of right zero-run).

Once you see that, the problem stops being about strings and starts being about runs. Instead of reading the string character by character, group it into runs of consecutive identical characters ("1000100" → 1, 000, 1, 00). Now you just need the biggest sum of two adjacent zero-runs sandwiching a one-run, added to your starting count of 1s.

To avoid special-casing the very first and last runs (what if a 1-block sits right at the edge with no bordering 0 on one side?), pad the string with a '1' sentinel on each end before computing runs. That guarantees every real run has neighbors on both sides, and it can never accidentally qualify as a "boxed in" candidate itself since it's part of the padding.

Brute Force Solution

Idea: For every maximal 1-block that has 0s on both sides, actually perform the swap — build the new string, count the 1s, and track the best result.

Algorithm: 1. Scan the string to find every boxed-in 1-group. 2. For each candidate, construct the resulting string (or simulate the count directly). 3. Track the maximum active count seen.

Advantages: Simple to reason about, easy to verify by hand, useful as a first correct answer in an interview to show you understand the problem before optimizing.

Disadvantages: Recomputing or rescanning for every candidate group is wasteful — each candidate potentially costs O(n) work, and there can be O(n) candidates.

Complexity: O(n²) time in the worst case (many candidate groups, each requiring an O(n) rescan), O(n) space for the working string.

Optimal Solution

The optimal approach never rewrites the string at all — it only counts.

  1. Pad the string with a '1' on each end. This guarantees the first and last runs are always padding, never real candidates, and removes all edge-case branching.
  2. Run-length encode the padded string using itertools.groupby, producing pairs like ('1', 1), ('0', 3), ('1', 1), ('0', 2), ('1', 1).
  3. Record the baseline: count of 1s in the original, unpadded string.
  4. Walk the runs, skipping the first and last (the sentinels). For every run of 1s, its gain is (previous run length) + (next run length), since neighboring runs are guaranteed to be 0-runs.
  5. Track the maximum gain across all interior 1-runs.
  6. Answer = baseline 1-count + best gain (0 if no valid 1-run exists).
Run Type Length Gain if traded
pad 1 1 — (sentinel)
1 0 3 —
2 1 1 3 + 2 = 5
3 0 2 —
pad 1 1 — (sentinel)

For "1000100": baseline = 2, best gain = 5, answer = 7. ✓

Python Code

from itertools import groupby


def max_active_section(s: str) -> int:
    """Return the max active (1) count after at most one boxed-in trade."""
    baseline_ones = s.count("1")

    padded = "1" + s + "1"
    runs = [(char, len(list(group))) for char, group in groupby(padded)]

    best_gain = 0
    for i in range(1, len(runs) - 1):  # skip sentinel runs at both ends
        char, length = runs[i]
        if char == "1":
            left_zeros = runs[i - 1][1]
            right_zeros = runs[i + 1][1]
            best_gain = max(best_gain, left_zeros + right_zeros)

    return baseline_ones + best_gain

Code Walkthrough

  • s.count("1") grabs the starting active count from the original string, before padding — the padding is only a scanning convenience, it shouldn't count toward the answer.
  • padded = "1" + s + "1" adds sentinel bumpers so every real 1-run has a bordering run on each side, eliminating edge-case checks.
  • groupby(padded) streams through the string and groups consecutive identical characters; wrapping each group in list(group) gives us its length.
  • The loop bounds range(1, len(runs) - 1) explicitly skip the first and last runs, which are always the sentinel 1s.
  • Inside the loop, we only care about 1-runs, since only trading a 1-block produces a gain. Its neighbors are guaranteed to be 0-runs (runs alternate by construction).
  • best_gain tracks the single largest sandwich found; the string itself is never mutated.

Dry Run

s = "00100"

  1. baseline_ones = 1.
  2. padded = "1001001" — wait, let's be precise: "1" + "00100" + "1" = "1001001".
  3. Runs: ('1',1), ('0',2), ('1',1), ('0',2), ('1',1).
  4. Skip index 0 and index 4 (sentinels).
  5. Index 1: ('0',2) — not a 1-run, skip.
  6. Index 2: ('1',1) — left neighbor ('0',2) = 2, right neighbor ('0',2) = 2 → gain = 4. best_gain = 4.
  7. Index 3: ('0',2) — skip.
  8. Answer = 1 + 4 = 5.

Matches the expected result: "00100" → trade the middle 1 → "11111" → 5.

Complexity Analysis

  • Time: O(n). s.count("1") is O(n), padding is O(n), groupby is a single linear pass producing O(n) total run elements, and the loop over runs is bounded by the number of runs, which is at most O(n). All steps are linear and sequential, so total time is O(n).
  • Space: O(n) in the worst case for the runs list (e.g., an alternating string like "010101..." produces O(n) runs), though in practice it's O(number of runs), which is often much smaller than n.
  • Why correct: Every possible trade's gain is fully determined by its two adjacent zero-run lengths, since the 1-block being traded contributes zero net change. Taking the max over all interior 1-runs finds the best possible trade in one pass, and the padding guarantees no valid candidate is missed or double-counted at the boundaries.

Alternative Solutions

Prefix sum approach: Precompute, for every index, how many consecutive 0s extend to the left and to the right. Then for each 1-block, look up the boundary values in O(1). This achieves the same O(n) time complexity but requires maintaining two additional O(n) arrays instead of one compact runs list. It's a reasonable alternative, but for this problem the run-length approach is simpler to implement and easier to reason about — there's no strong reason to prefer prefix sums unless you're already building them for another part of a larger solution.

Edge Cases

  • All zeros ("0000"): no 1-run exists at all, best_gain stays 0, answer is 0.
  • All ones ("1111"): every run is 1, but a 1-run adjacent to another 1-run (not 0) never happens after groupby, and there are no 0-runs to sandwich against — best_gain stays 0, answer equals the original count.
  • Single character ("0" or "1"): padding produces "101" or "111"; no interior boxed-in group exists, answer is the original count.
  • Empty string: s.count("1") is 0, padded string is "11", runs are just the two sentinels, no interior runs to check, answer is 0.
  • Multiple candidate 1-blocks of equal gain: max() naturally picks any one of them since we only need the best value, not which block produced it.
  • 1-block touching the true edge of the original string (e.g., "100"): after padding, this block is no longer adjacent to the sentinel directly — it still has a real 0-run neighbor on one side and the sentinel on the other, correctly treated as boxed in.

Common Mistakes

  1. Forgetting to pad the string, leading to index-out-of-range errors or incorrect special-casing for 1-blocks that touch the actual start or end of s.
  2. Counting 1s from the padded string instead of the original, which inflates the baseline by 2.
  3. Failing to skip the sentinel runs in the loop, which can incorrectly treat the padding itself as a candidate.
  4. Actually rewriting the string for each candidate (brute-force habit) instead of just tracking run lengths, turning an O(n) solution into O(n²).
  5. Assuming every 1-run has two 0-run neighbors without the padding trick, which breaks for 1-blocks at the true start or end of the string.
  6. Confusing "boxed in" with "any 1-run" — a 1-run adjacent to another 1-run can't happen after run-length encoding, but forgetting this alternation can lead to redundant checks.

Key Takeaways

Whenever a problem asks you to simulate an operation on a string of repeated values, ask whether the operation actually changes anything in the middle, or only at the boundaries. Collapsing a string into runs turns an O(n²) simulation into a single O(n) pass. The padding trick — adding sentinel values at both ends — is a reusable habit that eliminates an entire category of edge-case bugs, not just in this problem but anywhere you're scanning for boundaries.

# Maximize Active Section with Trade I — Python Solution

## Problem
Given a binary string `s`, you may perform exactly one operation: erase a
maximal group of `1`s that is bordered by `0`s on both sides, then fill the
entire span (the erased `1`s and both bordering `0`s) with `1`s. Return the
maximum number of active (`1`) positions achievable.

## Intuition
The block of `1`s you erase gets rebuilt as `1`s regardless — it contributes
zero net change. The only real gain comes from the `0`-runs bordering it,
which flip to active. So the problem reduces to finding the `1`-block whose
two adjacent zero-runs sum to the largest value.

## Approach
1. Pad `s` with a `'1'` sentinel on each end to remove edge-case branching.
2. Run-length encode the padded string with `itertools.groupby`.
3. Skip the two sentinel runs; for every interior `1`-run, compute
   `left_zero_run + right_zero_run`.
4. Track the maximum gain, add it to the original `1`-count.

## Python Solution
\`\`\`python
from itertools import groupby


def max_active_section(s: str) -> int:
    baseline_ones = s.count("1")

    padded = "1" + s + "1"
    runs = [(char, len(list(group))) for char, group in groupby(padded)]

    best_gain = 0
    for i in range(1, len(runs) - 1):
        char, length = runs[i]
        if char == "1":
            left_zeros = runs[i - 1][1]
            right_zeros = runs[i + 1][1]
            best_gain = max(best_gain, left_zeros + right_zeros)

    return baseline_ones + best_gain
\`\`\`

## Complexity
- **Time:** O(n) — one pass to count, one pass to run-length encode, one
  pass over the runs.
- **Space:** O(n) worst case for the runs list (bounded by number of runs).

📇 Flashcards

What pattern turns two separate 1-blocks into one big block?

Sandwich swap: flip a 0-gap between two 1-blocks (or an edge 0-run) to merge them into one contiguous run.

Time complexity of the optimal sandwich-swap approach?

O(n) — one pass to split into runs, one pass to check adjacent run triples.

Space complexity of the run-length approach?

O(n) worst case for the list of run lengths (or O(1) if scanning with rolling pointers instead of storing all runs).

Trickiest edge case in this problem?

A 0-run touching the array boundary (no 1-block on one side) — must handle it as a one-sided merge, not a full sandwich.

Brute force vs optimal: what's the key insight that avoids simulation?

Brute force tries every possible k-swap; optimal just breaks the string into runs and evaluates merges of adjacent runs directly, no simulation needed.

Interview follow-up: why break the string into runs first?

Runs reduce the problem to comparing a few adjacent group lengths instead of scanning bit-by-bit for every possible swap position.
