Introduction

Here's a question that trips up developers with years of experience: what does [10, 1, 3].sort() return in JavaScript? Most people confidently say [1, 3, 10]. The actual answer is [1, 10, 3].

This isn't an edge case buried in some obscure corner of the language. It's the default behavior of one of the most commonly used array methods in JavaScript. If you've never hit this bug, it's likely because your test data happened to be single-digit numbers, or already sorted, or you got lucky. In production, with real data, this bug shows up as scoreboards ranking wrong, prices sorting out of order, or dates landing in the wrong sequence.

Interviewers love this question because it tests something more valuable than memorizing a method signature. It tests whether you understand how a built-in function actually works under the hood, rather than assuming it does what its name implies. That distinction — assumption versus verification — is exactly what separates engineers who ship subtle bugs from engineers who catch them in code review.

By the end of this article, you'll understand exactly why sort() behaves this way, how to fix it permanently, and how to reason about similar "obvious but wrong" traps elsewhere in JavaScript.

Problem Overview

You're given an array of numbers. You call .sort() on it with no arguments, expecting the array to come back in ascending numeric order. Instead, the numbers come back ordered as if they were text.

Formally: Array.prototype.sort(), when called without a comparator function, converts every element to a string and sorts those strings in UTF-16 code unit order (essentially alphabetical/lexicographic order for typical ASCII digits).

This means: - 10 becomes "10" - 1 becomes "1" - 3 becomes "3"

And alphabetically, "1" comes before "10", which comes before "3" — because string comparison looks at characters left to right, and "1" is a shorter prefix match that sorts earlier than "3" regardless of what digits follow it.

The task: understand why this happens, and know the correct way to sort numbers so the result is numerically correct every time, regardless of array size or digit count.

Example

const nums = [10, 1, 3];
console.log(nums.sort());
// Output: [1, 10, 3]

Walking through why: - All three values are converted to strings: "10", "1", "3". - JavaScript compares them character by character, like a dictionary would. - "1" sorts before "10" and "3" because 1 < 3 as the first character, and "1" is a prefix of "10" (shorter strings sort first when one is a prefix of the other). - "10" sorts before "3" because the first character 1 is less than 3, even though 10 is numerically greater than 3.

Here's a more dramatic version of the same trap:

const nums = [100, 25, 3, 40];
console.log(nums.sort());
// Output: [100, 25, 3, 40]

That's not sorted at all by human intuition — because the comparison never looks at the numeric value, only at the string representation.

Intuition

The key insight an experienced engineer reaches for immediately: sort() needs to know how to compare two elements, and by default it assumes those elements are comparable as strings.

Think about why this default exists in the first place. sort() is a generic method — it works on arrays of strings, objects, mixed types, anything. JavaScript can't guess your intent for an arbitrary array, so the spec picked a universal fallback: convert everything to a string and compare lexicographically. That works reasonably well for arrays that are already strings (["banana", "apple"].sort() gives you the alphabetical order you'd expect), which is exactly why the bug hides so well — it only misbehaves once you're sorting numbers with more than one digit.

The moment you internalize that sort() is fundamentally string-first, the fix becomes obvious: don't let it guess. Tell it explicitly how to compare two elements by passing a comparator function. A comparator is just a function that takes two elements, a and b, and returns: - a negative number if a should come before b - a positive number if a should come after b - zero if they're equal

Once you supply that function, JavaScript stops converting anything to a string and just calls your function to decide order.

Brute Force Solution

Idea: Convert the array to numbers first, sort it as strings (accepting the bug), then try to "fix" the order some other way — for example, sorting the array manually with nested loops (a manual bubble sort or insertion sort) instead of relying on sort() at all.

Algorithm: 1. Loop over the array with an outer index i. 2. Loop over the array with an inner index j. 3. Compare arr[i] and arr[j] numerically using > or <. 4. Swap if out of order. 5. Repeat until the array is fully sorted.

function bubbleSort(arr) {
  const result = [...arr];
  for (let i = 0; i < result.length; i++) {
    for (let j = 0; j < result.length - i - 1; j++) {
      if (result[j] > result[j + 1]) {
        [result[j], result[j + 1]] = [result[j + 1], result[j]];
      }
    }
  }
  return result;
}

Advantages: - Correct numeric ordering, no string coercion trap. - Easy to understand and trace by hand.

Disadvantages: - Reinvents a wheel that the engine already provides, highly optimized, in native code. - O(n²) time complexity — far slower than the built-in sort on large arrays. - More code to write, test, and maintain for no real benefit.

Complexity: O(n²) time, O(n) space (for the copied array).

This "brute force" isn't really the recommended path — it exists mainly to illustrate that avoiding the bug doesn't require avoiding sort() altogether. The real optimal solution is much simpler.

Optimal Solution

Pass an explicit comparator function to sort(). This tells the engine exactly how to compare any two numbers, bypassing the string-conversion default entirely.

[10, 1, 3].sort((a, b) => a - b);

Here's why a - b works as a comparator:

a b a - b Meaning
1 10 -9 negative → a before b
10 3 7 positive → a after b
3 3 0 zero → equal, order unchanged

The comparator returns a negative number whenever a is numerically smaller, which tells sort() to place a first. This single arrow function replaces the flawed default comparator with a numerically correct one, and it costs nothing in performance — the underlying sort algorithm (typically some form of Timsort or an engine-optimized comparison sort) still runs in O(n log n), it's just now comparing the right thing.

For descending order, simply flip the subtraction:

[10, 1, 3].sort((a, b) => b - a); // [10, 3, 1]

Python Code

Since the underlying lesson — "don't rely on default comparison, be explicit" — applies universally, here's the equivalent correct approach in Python, where sorted() and list.sort() already compare numbers correctly by default, but the same principle applies when sorting complex objects:

def sort_numbers(nums: list[int]) -> list[int]:
    """Sort a list of numbers in ascending numeric order."""
    return sorted(nums, key=lambda x: x)


def sort_numbers_descending(nums: list[int]) -> list[int]:
    """Sort a list of numbers in descending numeric order."""
    return sorted(nums, key=lambda x: x, reverse=True)

Python's sorted() never falls into the JavaScript trap because it compares elements using their native type by default — numbers compare numerically, strings compare lexicographically, and mixing incompatible types raises a TypeError instead of silently coercing. That said, the key parameter exists for exactly the same reason JavaScript's comparator does: to make the comparison logic explicit when the default isn't what you want (for example, sorting strings that represent numbers, like ["10", "1", "3"], where you'd use key=int).

Code Walkthrough

  • sorted(nums, key=lambda x: x): key tells Python what value to compare each element on. Here it's an identity function, since the numbers are already directly comparable.
  • reverse=True: flips the final order without needing a different comparator, unlike JavaScript's flipped subtraction trick.
  • Both functions return a new list rather than mutating the input, which avoids the same class of "surprising side effect" bugs that sort()'s in-place mutation can cause in JavaScript.

Dry Run

Input: [10, 1, 3]

  1. sorted([10, 1, 3], key=lambda x: x) compares elements using Python's native < operator for integers.
  2. 1 < 3 < 10 numerically, so the result is [1, 3, 10].
  3. No string conversion ever happens — the comparison is numeric from the start.

Contrast with the JavaScript trap: [10, 1, 3].sort() gives [1, 10, 3] because it never compares numbers at all, only their string forms.

Complexity Analysis

  • Time: O(n log n) for both the JavaScript comparator-based sort and Python's sorted(), since both use efficient comparison-based sorting algorithms under the hood (Timsort in Python's CPython implementation, and an engine-specific comparison sort in V8/JavaScript engines).
  • Space: O(n) for Python's sorted(), since it returns a new list. JavaScript's sort() is in-place, so it's O(1) additional space (implementation-dependent, but generally no significant extra allocation for the array itself).
  • These are correct because comparison-based sorting has a proven lower bound of O(n log n) comparisons in the general case — no comparator, no matter how clever, can sort faster than that without exploiting extra structure in the data (like counting sort for bounded integers).

Alternative Solutions

If you know your numbers are non-negative integers within a bounded range, a counting sort or radix sort can achieve O(n + k) time, where k is the range of values. This trades the generality of comparison-based sorting for speed, and only makes sense when you have strong guarantees about your input (for example, sorting exam scores from 0–100).

For typical interview and production scenarios, though, the comparator-based approach is preferred: it's general, correct for any numeric range, and doesn't require assumptions about the data.

Edge Cases

  • Empty array: [].sort((a, b) => a - b) returns [] safely — no comparator calls happen.
  • Single element: [5].sort((a, b) => a - b) returns [5] — trivially sorted.
  • Duplicate values: [3, 3, 1].sort((a, b) => a - b) returns [1, 3, 3] — the comparator correctly returns 0 for equal values, preserving stability.
  • Negative numbers: [-10, 5, -3].sort((a, b) => a - b) returns [-10, -3, 5] — subtraction handles negatives correctly, unlike string comparison, which would place "-10" in a confusing position relative to "5".
  • Very large numbers: For numbers near Number.MAX_SAFE_INTEGER, subtraction-based comparators can theoretically overflow floating point precision in extreme cases; safer to use explicit if/else returns (a < b ? -1 : a > b ? 1 : 0) when working near those limits.

Common Mistakes

  1. Forgetting the comparator entirely, assuming sort() "just works" for numbers.
  2. Writing (a, b) => a > b as the comparator — this returns a boolean, not a signed number. It often appears to work by coincidence in some engines but isn't spec-compliant and can produce unstable results.
  3. Mutating an array you didn't mean to change, since sort() sorts in place and returns the same reference — always copy first ([...arr].sort(...)) if you need the original preserved.
  4. Assuming sort() is stable across all engines without verifying — modern engines guarantee stability per spec, but it's worth knowing rather than assuming for older environments.
  5. Using string-based sort logic on numeric strings (like ["10", "2", "1"]) without converting to numbers first inside the comparator ((a, b) => Number(a) - Number(b)).

Key Takeaways

  • JavaScript's Array.prototype.sort() defaults to string comparison — always pass a comparator when sorting numbers.
  • A correct numeric comparator returns negative, zero, or positive based on numeric difference: (a, b) => a - b.
  • This bug is a reminder to verify assumptions about built-in methods rather than trusting names or habits from other languages.
  • Python's sorted() avoids this specific trap by comparing native types directly, but the same principle of being explicit with key still applies for custom orderings.

# JavaScript sort() Trap: The Default Comparator Problem

## Problem
`[10, 1, 3].sort()` does not return `[1, 3, 10]`. It returns `[1, 10, 3]`.

`Array.prototype.sort()` called without arguments converts every element to a
string and compares them lexicographically (like dictionary order), not
numerically.

## Intuition
`sort()` is generic — it works on any array type, so its default fallback is
string comparison, which happens to be correct for arrays of strings but
silently wrong for arrays of numbers with differing digit counts.

## Approach
Pass an explicit comparator function. It returns:
- negative → first argument sorts first
- positive → first argument sorts after
- zero → equal, order unchanged

```js
[10, 1, 3].sort((a, b) => a - b); // [1, 3, 10]

Python Solution

def sort_numbers(nums: list[int]) -> list[int]:
    """Sort a list of numbers in ascending numeric order."""
    return sorted(nums, key=lambda x: x)


def sort_numbers_descending(nums: list[int]) -> list[int]:
    """Sort a list of numbers in descending numeric order."""
    return sorted(nums, key=lambda x: x, reverse=True)
