Getting Started: Installing Python and Running Code #shorts

General

Introduction

Every developer remembers the first time they installed Python and then sat staring at a terminal, unsure what to type next. It seems like it should be trivial — you installed the language, so surely just typing python should work everywhere, right? Not quite.

This is one of those deceptively small details that trips up almost every beginner, and honestly, a fair number of experienced developers who switch operating systems for the first time. It rarely comes up as a formal interview question, but it absolutely shows up in real onboarding situations: a new hire joins a team, pulls the setup docs, runs python script.py, and gets command not found. Suddenly they're debugging their environment instead of their code — which is a frustrating way to start.

Understanding why this happens teaches you something deeper than "which command to type." It teaches you how operating systems handle executables, how Python versioning actually works under the hood, and why tooling differs so much between Windows, macOS, and Linux. That foundation pays off every time you set up a new machine, onboard a teammate, or debug a "works on my machine" issue.

Problem Overview

Here's the situation: you've just installed Python on your computer. You open a terminal and want to run a script. The question is simple on the surface — which command actually executes your Python file?

The intuitive answer is python. But depending on your operating system and how Python was installed, that single command might:

  • Work perfectly
  • Not exist at all
  • Point to an old Python 2 installation (on older systems)
  • Point to a completely different program

The real answer depends on your OS:

  • Windows: python usually works, but Windows also ships a special launcher called py, which is often the more reliable choice.
  • macOS: python may not point to anything, or may point to a system-level Python 2 leftover. python3 is almost always the safe, correct command.
  • Linux: Similar story to macOS — python3 is the dependable choice on most modern distributions.

So the "trick" in this lesson isn't really a trick — it's a reminder that installing a language and knowing how your OS exposes that language on the command line are two separate skills.

Example

Let's walk through three scenarios, one per OS, so the difference is concrete.

Scenario 1: Windows

C:\Users\dev> python hello.py
Hello, World!

This works fine on most Windows installations because the official Python installer registers python in your system PATH by default.

But Windows also gives you this option:

C:\Users\dev> py hello.py
Hello, World!

Scenario 2: macOS

$ python hello.py
zsh: command not found: python

$ python3 hello.py
Hello, World!

On modern macOS, python is frequently unavailable (or reserved), so python3 is the command that actually works.

Scenario 3: Linux

$ python hello.py
Command 'python' not found, did you mean:
  command 'python3' from deb python3

$ python3 hello.py
Hello, World!

Same pattern as macOS. The system is explicitly nudging you toward python3.

Intuition

Think about what "installing Python" actually means to your operating system. It's not magic — it's placing an executable file somewhere on disk and (ideally) registering that location in your system's PATH, the list of directories your terminal searches when you type a command.

The naming of that executable is a convention, not a law of nature. Python's maintainers made a deliberate choice years ago, during the Python 2 to Python 3 transition, to avoid silently overwriting the python command on Unix-like systems. Many machines still had Python 2 installed as part of the OS itself (older macOS and Linux distributions relied on it for internal scripts). Overwriting python with Python 3 could have broken system tools.

So the community landed on a convention: keep python pointing at whatever legacy version might exist, and use python3 to explicitly mean "the Python 3 interpreter." Windows didn't have this legacy Python 2 baggage baked into the OS in the same way, so the official installer was free to bind python directly to whatever version you installed.

Windows then added something extra: the py launcher. Instead of picking one fixed executable, py reads a small marker (or defaults to the latest installed version) and picks the correct interpreter for you — even if you have multiple Python versions installed side by side. That's genuinely useful once you're juggling Python 3.9 for one project and Python 3.12 for another.

Once you see it this way, the "trick" disappears. It's not that Python behaves inconsistently — it's that each OS exposes the interpreter through a slightly different naming convention, shaped by its own history.

Brute Force Solution

There isn't really a "brute force algorithm" here since this isn't a data structures problem — but there is a brute-force mental model beginners often use, and it's worth examining why it fails.

Idea: Assume python always works, everywhere, because that's the "obvious" command name.

Approach: 1. Install Python. 2. Type python in the terminal. 3. Assume it will run your script.

Advantages: - Simple to remember. - Works out of the box on many Windows machines.

Disadvantages: - Fails silently or loudly on macOS and Linux. - Can point to the wrong Python version if multiple are installed. - Leads to confusing errors like ModuleNotFoundError when pip and python are quietly pointing to different installations.

Time and space complexity: Not applicable — this is an environment configuration issue, not a computational one. But the "cost" here is developer time lost to debugging, which is arguably worse than an inefficient algorithm.

Optimal Solution

The optimal approach is to check your specific OS and use the command that's guaranteed to work, rather than assuming.

OS Reliable Command Why
Windows py Official launcher, auto-selects installed version
Windows (alt) python Works if PATH was configured during install
macOS python3 Avoids legacy/reserved python naming conflicts
Linux python3 Same reasoning as macOS

A more robust habit, regardless of OS, is to verify before you assume:

python --version
python3 --version
py --version

Whichever command returns a valid Python 3.x version number is the one to use consistently for that machine. This small verification step removes the guesswork entirely and prevents the classic "which Python is pip install even using" confusion down the line.

Python Code

Since this lesson is about environment setup rather than an algorithm, the "code" here is a small diagnostic script you can run to confirm your setup — useful for onboarding docs or a first-day setup guide.

import shutil
import subprocess


def find_python_commands():
    """Check which Python launcher commands are available on this machine."""
    candidates = ["python", "python3", "py"]
    available = []

    for cmd in candidates:
        path = shutil.which(cmd)
        if path is None:
            continue

        result = subprocess.run(
            [cmd, "--version"],
            capture_output=True,
            text=True,
        )
        version = result.stdout.strip() or result.stderr.strip()
        available.append((cmd, path, version))

    return available


if __name__ == "__main__":
    found = find_python_commands()

    if not found:
        print("No Python command found on PATH. Check your installation.")
    else:
        print("Available Python commands on this system:\n")
        for cmd, path, version in found:
            print(f"  {cmd:<8} -> {version}  ({path})")

Code Walkthrough

  • shutil.which(cmd) checks whether a given command exists anywhere on your system PATH, returning the full path if found, or None if not. This mirrors exactly what your terminal does when you type a command.
  • We loop over the three realistic candidates: python, python3, and py, since those cover Windows, macOS, and Linux.
  • subprocess.run([cmd, "--version"], ...) actually invokes each command to confirm it runs and to capture its reported version — some tools print the version to stdout, others to stderr, so we check both.
  • The script prints a clean summary table showing every valid Python entry point available on the machine, along with its file path and version.

Dry Run

Suppose you run this script on a Windows machine with Python 3.11 installed via the official installer.

  1. shutil.which("python") finds C:\Users\dev\AppData\Local\Programs\Python\Python311\python.exe.
  2. shutil.which("python3") returns None — Windows doesn't register this by default.
  3. shutil.which("py") finds C:\Windows\py.exe, the official launcher.
  4. For each found command, we run --version and capture the output, e.g. Python 3.11.4.
  5. Final output:
Available Python commands on this system:

  python   -> Python 3.11.4  (C:\Users\dev\AppData\Local\Programs\Python\Python311\python.exe)
  py       -> Python 3.11.4  (C:\Windows\py.exe)

This confirms both python and py work identically on this machine — but on a Mac, the same script would likely show only python3 populated.

Complexity Analysis

This isn't an algorithmic problem in the traditional Big-O sense, but it's worth reasoning about the diagnostic script:

  • Time complexity: O(1) relative to input size — we check a fixed, small set of three candidate commands regardless of machine configuration. Each subprocess.run call has real-world latency (process spawn time), but that's constant per command, not something that scales with any "n."
  • Space complexity: O(1) — we store at most three tuples of command, path, and version string.

These are correct because the work done is bounded by the fixed list of candidate executable names, not by any variable-sized input.

Alternative Solutions

An alternative to checking manually is using tools designed specifically for managing multiple Python versions:

  • pyenv (macOS/Linux) and pyenv-win (Windows) let you install and switch between multiple Python versions cleanly, without relying on OS-level naming quirks.
  • Virtual environments (venv or virtualenv) sidestep the global command confusion entirely by creating an isolated Python and pip binary scoped to your project.

Comparison: The diagnostic script above is great for quickly understanding what's currently on your system. Tools like pyenv or virtual environments are better long-term solutions once you're working across multiple projects with different Python version requirements. Think of the script as a flashlight to see what you have, and pyenv/venv as the actual fix for managing it.

Edge Cases

  • No Python installed at all: All three commands return None from shutil.which. The script correctly reports nothing found.
  • Python 2 leftover on the system: python might exist but point to Python 2.7. The version check catches this — always confirm the version string, don't just confirm the command exists.
  • Multiple Python versions installed on Windows: py will pick a default version, but you can target a specific one with py -3.9 or py -3.12, which is worth knowing when working across legacy projects.
  • PATH not updated after install: Especially common on Windows if "Add Python to PATH" was unchecked during installation — python and py might both fail even though Python is technically installed.
  • Corporate/restricted environments: Some managed machines alias python to a wrapper script or restrict subprocess calls; the diagnostic script should still run safely since it only invokes --version.

Common Mistakes

  1. Assuming python works identically across every operating system.
  2. Not checking python --version after installation to confirm which version is actually active.
  3. Installing Python without adding it to PATH, then wondering why the terminal can't find it.
  4. Mixing pip and pip3 inconsistently, ending up installing packages into a different Python environment than the one actually being run.
  5. Ignoring the py launcher on Windows entirely, missing out on its convenient version-targeting (py -3.10).
  6. Not realizing that IDEs (VS Code, PyCharm) can be configured to use a completely different Python interpreter than what your terminal defaults to, causing "it works in my editor but not in the terminal" confusion.

Key Takeaways

  • Installing Python and knowing how to invoke Python are two separate skills — don't assume they're the same.
  • python, python3, and py exist because of historical decisions around the Python 2 to 3 transition and OS-specific tooling choices.
  • Always verify with --version rather than assuming a command works.
  • On Windows, the py launcher is often the most reliable and flexible option.
  • Small environment details like this are exactly the kind of thing that silently costs new developers hours — understanding the "why" prevents that friction permanently.
