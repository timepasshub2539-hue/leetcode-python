8. Introduction

Every developer eventually meets the XOR swap trick. It shows up in interview prep threads, in old C codebases, and in "cool tricks" lists: swap two values without a temporary variable, using nothing but three XOR operations. It feels like a magic spell — no extra memory, no third box, just bitwise sleight of hand.

But this trick hides one of the most instructive bugs in all of programming. It's not a syntax error. It's not a typo. It's a logical failure that only appears on a specific edge case, and it silently corrupts data instead of crashing loudly.

Interviewers love this topic because it tests more than "do you know XOR." It tests whether you understand aliasing — the situation where two names or two indices refer to the same underlying memory. It tests whether you reason about edge cases before trusting clever code. And it tests judgment: knowing when a micro-optimization is worth the risk and when the boring solution is simply better engineering.

If you can explain why the XOR swap breaks and when to avoid it, you demonstrate exactly the kind of low-level reasoning that separates someone who memorizes tricks from someone who understands them. Let's build that understanding from the ground up.

9. Problem Overview

The task sounds trivial: given two variables (or two array elements), exchange their values so that the first holds what the second held, and vice versa.

The classic beginner solution uses a temporary variable:

temp = a
a = b
b = temp

Three assignments, one extra variable. Clear, correct, done.

The "clever" alternative claims you can do this with zero extra storage by exploiting a property of the XOR (^, exclusive or) operator. XOR has a beautiful mathematical structure: a number XORed with itself is 0, and XOR is reversible. That reversibility is what makes swapping without a temp seem possible.

The catch — and the real problem we're solving today — is this: the XOR swap only works when the two operands are stored in different locations. The moment they alias the same location (for example, you call swap(arr, i, j) with i == j), the trick doesn't just fail to swap. It zeroes the value out permanently. Your data is gone, with no error message to warn you.

So the real question isn't "how do I swap two numbers?" It's "why does this specific implementation break, and how do I write swap code that is safe in every case?"

10. Example

Let's make it concrete with two scenarios.

Scenario A — different variables (works correctly):

Step Operation a b
Start — 5 9
1 a = a ^ b 12 9
2 b = a ^ b 12 5
3 a = a ^ b 9 5

Result: a = 9, b = 5. Swapped correctly.

Scenario B — same location (the bug):

Imagine swap(arr, i, j) where i == j, and arr[i] = 7. Both x and y below refer to the same slot:

Step Operation Value in the shared slot
Start — 7
1 slot = slot ^ slot 7 ^ 7 = 0
2 slot = slot ^ slot 0 ^ 0 = 0
3 slot = slot ^ slot 0 ^ 0 = 0

Result: the value 7 is destroyed and replaced with 0. The number didn't get swapped with itself (which should be a harmless no-op) — it got erased. That is the bug the quiz answer "B" points to: when i == j, x ^ x is 0, and your value is gone forever.

11. Intuition

Here's how an experienced engineer discovers this bug without needing anyone to point it out.

XOR swapping relies on the identity v ^ v == 0. That's the engine of the whole trick — it's what lets the second step "cancel out" one value and recover the other. When a and b are separate, cancellation is helpful: it strips a back out so b can absorb the original a.

But the same identity that makes the trick work is a landmine when both operands are the same storage. The first line, a = a ^ b, is supposed to fold both values into a. When a and b are the same slot, you're computing slot ^ slot, which is 0 — and you've just overwritten your only copy of the value with zero. There is no longer any b sitting somewhere else to recover it from. Every subsequent XOR just churns zeros.

The mental red flag is this: any in-place algorithm that writes to a location before reading everything it needs is fragile under aliasing. A seasoned engineer sees "swap without temp using the destination as scratch space" and immediately asks: what if source and destination are the same? That single question surfaces the bug before a single test runs.

The temp-variable version doesn't have this problem because it never destroys a value before copying it somewhere safe. temp = a preserves the value first. Even if a and b are the same slot, you end up assigning the value back to itself — a harmless no-op.

12. Brute Force Solution

Idea: Use a temporary variable to hold one value while you overwrite it.

Algorithm: 1. Copy a into temp. 2. Assign b into a. 3. Assign temp into b.

def swap_with_temp(arr, i, j):
    temp = arr[i]
    arr[i] = arr[j]
    arr[j] = temp

Advantages: - Correct in every case, including i == j (it degrades to a safe no-op). - Instantly readable — any developer understands it at a glance. - Works for any data type, not just integers.

Disadvantages: - Uses one extra variable's worth of memory (a few bytes — almost always irrelevant).

Time complexity: O(1) — three assignments. Space complexity: O(1) — one temporary.

Calling this "brute force" is generous; it's actually the correct baseline. The XOR version is the risky "optimization."

13. Optimal Solution

The XOR swap is the "no extra space" approach — genuinely elegant when used correctly, and worth understanding because it teaches how XOR's reversibility works.

The three-step recipe:

a = a ^ b     # a now holds the combined bits of both
b = a ^ b     # b recovers the ORIGINAL a
a = a ^ b     # a recovers the ORIGINAL b

Why each step works, bit by bit:

Step Expression What it produces
1 a = a ^ b a becomes A ^ B (a mix carrying enough info to recover either)
2 b = a ^ b = (A ^ B) ^ B B cancels, leaving A → b now holds original A
3 a = a ^ b = (A ^ B) ^ A A cancels, leaving B → a now holds original B

The magic is XOR self-cancellation: X ^ X = 0 and X ^ 0 = X. Step 2 removes the B component; step 3 removes the A component. Perfect swap — as long as a and b are distinct storage.

The truly optimal solution in production, though, recognizes the aliasing trap and guards against it — or just uses the temp version. If you insist on XOR swap, protect it:

def xor_swap(arr, i, j):
    if i != j:                # guard: skip when both point to the same slot
        arr[i] ^= arr[j]
        arr[j] ^= arr[i]
        arr[i] ^= arr[j]

That single if i != j check turns a data-destroying trick into a safe operation.

14. Python Code

def swap_with_temp(arr, i, j):
    """Swap two elements safely. Correct even when i == j."""
    arr[i], arr[j] = arr[j], arr[i]  # Pythonic, uses tuple packing under the hood


def xor_swap(arr, i, j):
    """XOR swap with an aliasing guard.

    Without the i != j check, xor_swap(arr, i, i) would zero out arr[i].
    """
    if i != j:
        arr[i] ^= arr[j]
        arr[j] ^= arr[i]
        arr[i] ^= arr[j]


def xor_swap_buggy(arr, i, j):
    """The classic BUGGY version — kept for demonstration only. Do not use."""
    arr[i] ^= arr[j]   # when i == j, this is arr[i] ^= arr[i] -> 0
    arr[j] ^= arr[i]
    arr[i] ^= arr[j]


if __name__ == "__main__":
    # Correct behavior on distinct indices
    data = [5, 9]
    xor_swap(data, 0, 1)
    assert data == [9, 5], data

    # The guarded version is safe on the same index
    same = [7]
    xor_swap(same, 0, 0)
    assert same == [7], "guarded swap must leave the value untouched"

    # Proof of the bug: unguarded XOR swap destroys the value
    broken = [7]
    xor_swap_buggy(broken, 0, 0)
    assert broken == [0], "buggy version zeroes the value — this is the bug"

    print("All checks passed. The guard is what saves you.")

Note that in idiomatic Python you'd almost always write arr[i], arr[j] = arr[j], arr[i]. The interpreter evaluates the right side into a tuple first, so it is aliasing-safe and needs no temp of your own. The XOR version exists mainly to teach the underlying mechanics.

15. Code Walkthrough

  • swap_with_temp / tuple swap — arr[i], arr[j] = arr[j], arr[i] builds the tuple (arr[j], arr[i]) fully before assigning. Because both reads happen before any write, i == j is harmless.
  • xor_swap guard — if i != j: is the entire fix. It ensures we only run the XOR sequence when the two slots are genuinely different storage.
  • arr[i] ^= arr[j] — folds both values' bits into slot i. This is the step that would destroy the value if i == j.
  • arr[j] ^= arr[i] — uses the folded value to recover the original arr[i] into slot j.
  • arr[i] ^= arr[j] — uses the folded value again to recover the original arr[j] into slot i.
  • The __main__ self-check — three assertions prove the guarded version is correct, prove it's safe on the same index, and prove the bug exists in the unguarded variant by showing the value becomes 0.

16. Dry Run

Let's dry-run xor_swap_buggy(broken, 0, 0) with broken = [7].

Binary of 7 is 0111.

  1. arr[0] ^= arr[0] → 0111 ^ 0111 = 0000 → arr[0] is now 0.
  2. arr[0] ^= arr[0] → 0000 ^ 0000 = 0000 → still 0.
  3. arr[0] ^= arr[0] → 0000 ^ 0000 = 0000 → still 0.

Final: broken = [0]. The 7 is unrecoverable — there was never a second copy to XOR back from. This is precisely why the quiz's answer B is the bug: x ^ x collapses to zero on the very first step.

Now compare the guarded call xor_swap(same, 0, 0): the if i != j is if 0 != 0, which is False, so the body is skipped entirely. same stays [7]. Safe.

17. Complexity Analysis

  • Time: O(1) for every version. Swapping is a constant number of operations regardless of input size — three XORs, or one tuple assignment.
  • Space: O(1) for all versions. The temp/tuple approach uses a tiny fixed amount of extra memory; XOR uses none beyond the operands. In practice this "savings" is a few bytes and is completely negligible on any modern machine.

Why these are correct: there are no loops and no recursion. The work does not grow with the size of the array — you always touch exactly two slots. The much-hyped memory advantage of XOR swap is real only in the narrowest embedded contexts, and even there the correctness risk usually outweighs it.

18. Alternative Solutions

Pythonic tuple swap (recommended):

arr[i], arr[j] = arr[j], arr[i]

This is the clear winner in Python. It's readable, safe under aliasing, works for any type (strings, objects, floats — not just integers), and the interpreter handles the temporary for you.

Arithmetic swap (another "no temp" trick):

a = a + b
b = a - b
a = a - b

This has the same class of bug plus an extra one: it also breaks when a and b alias, and it can overflow in fixed-width integer languages. In Python, integers are arbitrary precision so overflow isn't an issue, but the aliasing problem remains. Same lesson, same fix.

Comparison:

Approach Extra memory Safe on aliasing? Works on non-integers? Readable?
Tuple swap ~O(1) ✅ Yes ✅ Yes ✅ Excellent
Temp variable O(1) ✅ Yes ✅ Yes ✅ Excellent
XOR (guarded) O(0) ✅ Yes ❌ Ints only ⚠️ Needs comment
XOR (unguarded) O(0) ❌ No ❌ Ints only ❌ Trap

19. Edge Cases

  • Same index / aliasing (i == j): The headline bug. Unguarded XOR destroys the value. Always the first case to test.
  • Empty input: Calling swap on an empty array with any index raises IndexError — validate indices before swapping if input is untrusted.
  • Single element with valid same index: swap(arr, 0, 0) must be a no-op, not a wipe.
  • Duplicate values (arr[i] == arr[j] but i != j): Works fine — the indices differ, so XOR still cancels correctly. Equal values are not the same as equal indices.
  • Minimum / maximum integers: In Python, arbitrary-precision integers mean no overflow; the arithmetic-swap trick would be dangerous here in C/Java, but XOR is bit-safe regardless of magnitude.
  • Negative numbers: XOR operates on the two's-complement bit patterns and swaps them correctly, as long as the aliasing guard holds.

20. Common Mistakes

  1. Assuming i == j is impossible. In real sort and partition code, i == j happens constantly (a pivot equal to a boundary, a fixed point). This is the mistake the quiz targets.
  2. Confusing equal values with equal indices. Two different slots holding the same number swap fine. Only shared storage is the problem.
  3. Using XOR swap on non-integers. ^ isn't defined for floats, strings, or objects. The trick is integer-only.
  4. Porting the arithmetic swap to C/Java without thinking about overflow. a + b can overflow fixed-width integers and silently corrupt results.
  5. Reaching for the "clever" trick to save memory that doesn't matter. A few bytes are never worth a data-loss bug. Optimize for correctness and readability first.
  6. Trusting a trick you can't dry-run from memory. If you can't trace x ^ x on the spot, you shouldn't ship it in a hot path.

23. Key Takeaways

  • The XOR swap works by folding two values together, then cancelling out one at a time using x ^ x = 0.
  • It has a fatal aliasing bug: when both operands are the same storage (i == j), the first XOR zeroes the value permanently.
  • A single if i != j: guard makes it safe — but at that point, a temp variable or tuple swap is simpler and safer.
  • Clever, memoryless tricks trade a negligible byte for a real correctness risk. In modern engineering, that's a bad trade.
  • The deeper lesson: any in-place operation that overwrites storage before reading everything it needs is fragile under aliasing. Always ask "what if source and destination are the same?"

# XOR Swap: The Same-Variable Bug

Swapping two values without a temporary variable using XOR — and the
aliasing bug that makes it dangerous.

## Problem

Swap two values in place. The classic "no temp variable" solution uses
three XOR operations. It works correctly when the two operands live in
**different** storage, but breaks silently when they alias the same
location (e.g. `swap(arr, i, j)` with `i == j`).

## Intuition

XOR swap relies on two properties:
- `x ^ x == 0` (self-cancellation)
- XOR is reversible

It folds both values together, then cancels one out at a time to recover
the other. But when both operands are the **same** storage, the first
step computes `x ^ x == 0` — overwriting the only copy of the value with
zero before anything can recover it. The value is destroyed, not swapped.

## Approach

| Step | Expression        | Result                     |
|------|-------------------|----------------------------|
| 1    | `a = a ^ b`       | `a` = A ^ B                |
| 2    | `b = a ^ b`       | `b` = original A           |
| 3    | `a = a ^ b`       | `a` = original B           |

Fix: guard with `if i != j` — or just use a temp / tuple swap, which is
safe in every case and works on any type.

## Python Solution

```python
def xor_swap(arr, i, j):
    """XOR swap with an aliasing guard (safe when i == j)."""
    if i != j:
        arr[i] ^= arr[j]
        arr[j] ^= arr[i]
        arr[i] ^= arr[j]


def swap(arr, i, j):
    """Recommended: Pythonic, safe, works on any type."""
    arr[i], arr[j] = arr[j], arr[i]


if __name__ == "__main__":
    d = [5, 9]
    xor_swap(d, 0, 1)
    assert d == [9, 5]

    same = [7]
    xor_swap(same, 0, 0)      # guarded -> no-op
    assert same == [7]

    print("OK")

Complexity

  • Time: O(1)
  • Space: O(1)
